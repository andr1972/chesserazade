# 0.7 — Zobrist hashing + transposition table

Tagged as **v0.7.0**.

The search gets memory. Positions reached by different move
orders are now recognized as equal — a sub-tree searched once
doesn't have to be searched again. This also addresses the
user's original note from the handoff about
"cechowanie ruchów A,B,C = C,B,A": a Zobrist-keyed TT is the
classical way to make the engine see through move-order
transpositions.

## What was added

### Zobrist keys
`include/chesserazade/zobrist.hpp` + `src/board/zobrist.cpp`.

A fixed 64-bit random-constant table indexed by:
  * piece-type × color × square  (12 × 64 = 768 values)
  * castling-rights bitmask      (16 values)
  * en-passant *file*            (8 values)
  * side-to-move                 (1 value XORed when black to move)

The constants are initialized from a fixed seed
(`0xCE55E5A2ADE12345`) so two runs of the engine produce
identical keys — a precondition for persisting a 0.8+ opening
book.

`Board` gains a pure-virtual `zobrist_key() const`. The mailbox
implementation maintains the key incrementally through a small
`place()` helper that XOR-updates on every piece write inside
`make_move`; `unmake_move` restores the pre-move key from the
history snapshot. EP / castling / side-to-move contributions
are XORed in/out explicitly at the entry/exit of `make_move`.

### TranspositionTable
`include/chesserazade/transposition_table.hpp` +
`src/search/transposition_table.cpp`.

Fixed-size open-addressing-style table — index = `key & mask`,
where `mask` is `size - 1` on a power-of-two table. Each entry:

```
TtEntry { key, move, score (int16), depth (u8), flags (u8) }
```

`flags` packs the bound type (`Exact`, `Lower`, `Upper`) in the
low 2 bits and a 6-bit age in the high 6 bits. Replacement
scheme:
```
replace if   slot is empty
          OR slot key == new key       (refresh)
          OR slot age != current age   (stale generation)
          OR new depth >= slot depth   (deeper analysis wins)
```

### Search integration
The negamax core in `src/search/search.cpp` gains an optional
`TranspositionTable*` parameter:

* **Probe** at non-root nodes: a hit at `depth >= remaining
  depth` can return early, respecting the entry's bound
  (`Exact` always; `Lower` if `score >= beta`; `Upper` if
  `score <= alpha`).
* **Store** on beta-cutoff (bound = `Lower`) and on normal
  return (`Exact` if alpha was raised, `Upper` otherwise).

Mate scores are stored in a ply-relative form (`score ± ply` on
store, reversed on probe) so a cached entry still means "mate
in N plies from here" when the same position is encountered at
a different root distance.

The root node does **not** take TT cutoffs — we always complete
the iteration so `SearchResult.principal_variation` reflects a
real tree walk.

### `SearchResult` / CLI
`SearchResult` gains `tt_probes` and `tt_hits`. The `solve`
CLI now prints:
```
best:  Nc3  (b1c3)
score: +150 cp
depth: 5
pv:    3. Nc3 Nf6 4. Bb5 Nb4 5. Nxe5
nodes: 335221
tt:    335216 probes, 6186 hits (1.8%)
time:  514 ms
```

Hit rates in 0.7 are modest (~2% at depth 5 on a no-ordering,
no-quiescence search) — without move ordering, most TT probes
miss the *best* entry because we haven't yet searched the
subtree where that entry was stored. Hit rates will jump once
move ordering (0.8) steers the search toward the already-known
best moves.

## Acceptance criteria (HANDOFF §9 0.7)

- [x] Zobrist keys maintained incrementally in `make_move` /
      `unmake_move`.
- [x] Fixed-size transposition table with a replacement scheme
      based on age + depth.
- [x] TT probes and stores wired into the search.
- [x] Addresses move-order-independent transpositions: a
      transposition test (`1.Nf3 Nc6 2.Nc3 Nf6` vs
      `1.Nc3 Nf6 2.Nf3 Nc6`) shows equal Zobrist keys.
- [x] Measurable node reduction on a tactical position at
      depth 4 — test
      `Search + TT: same best move, fewer nodes`.
- [x] TT hit rate logged — `solve` prints `tt: P probes, H hits
      (R%)`.

## What is *not* in 0.7

- **Move ordering.** Without it, TT hit rate is small and the
  TT *move* is stored but not yet used. 0.8 will use the TT
  move as the first-tried move, at which point hit rates jump
  dramatically.
- **Quiescence search.** 0.8.
- **PVS / null-move / late-move reductions.** Out of scope for
  the 0.x line (except where they'd be obvious teaching
  material).
- **`--hash` flag** to control TT size. Default is a 1M-entry
  (~16 MiB) table. A flag is a one-commit addition when a user
  needs it.
- **Clearing the TT on position change in the REPL.** Not yet —
  currently each `solve` invocation gets a fresh table. When
  the REPL calls into Search in a future version the
  clear-on-load discipline will matter.

## Known deviations

- The TT-move is stored but never used for ordering (ordering
  is 0.8). This is deliberate: storing it now means no data
  migration later.
- Node count grows monotonically across iterative-deepening
  depths in `SearchResult`; the reduction from TT shows up as
  a lower total relative to the non-TT run, not as a per-depth
  percentage.
