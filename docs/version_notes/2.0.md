# 2.0 — Search hardening: NMP and LMR

The 1.x line covered the "classical basics" of an alpha-beta
search engine — α-β pruning, iterative deepening, transposition
table, MVV-LVA + killers, quiescence. 2.x opens with two
techniques that turn that baseline into a competitive search:
*null-move pruning* (NMP) and *late move reductions* (LMR).
Both reduce the effective branching factor by selectively
*not* doing full-depth work on subtrees that almost certainly
do not change the answer.

NMP and LMR are the first two members of the 2.x feature set;
later pages add aspiration windows, history heuristic, PVS, and
check extensions.

## 2.0.1 — Null-move pruning

**Idea.** Before searching this node's real moves, ask a
hypothetical question: *if the side to move passed (made no
move at all) and gave the opponent two moves in a row, would
the opponent still fail to push the score above β?* If even
with that free move the opponent cannot beat β, the position
is already so good for the side to move that searching the
real moves can only confirm a cutoff that's already proven.
We return a β-cutoff immediately and skip generating moves at
all.

The technique is unsound in zugzwang positions — where every
move is strictly worse than passing — so it needs careful
guards. With those guards in place it typically prunes 30–60%
of nodes at middle game depths.

**Where.** `src/search/search.cpp:507-562`, inside `negamax`,
between the TT probe and the legal-move generation.

**Mechanism.**

```cpp
constexpr int NMP_R = 3;
if (allow_null && !disable_alpha_beta && ply > 0 && depth >= 3
    && !is_in_check(...) && has_non_pawn_material(...)
    && static_eval >= beta) {
    board.make_null_move();
    NegamaxResult nr = negamax(
        board, depth - NMP_R - 1, ply + 1,
        -beta, -beta + 1,        // null-window around β
        ..., /*rec=*/nullptr,    // shadow search, no recorder
        /*allow_null=*/false);   // never two nulls in a row
    board.unmake_null_move();
    if (-nr.score >= beta) return {-nr.score, /*exact=*/false};
}
```

The null move is implemented at `Board::make_null_move()` /
`unmake_null_move()` — only side-to-move and en-passant state
flip, the Zobrist key is XOR'd accordingly, and a small undo
record is pushed. No real move is played.

The reduced search runs at `depth − R − 1` (for R = 3, three
plies shallower than a normal child) with a *zero-width
window* `[β − 1, β]`: we don't need the actual score, only
the boolean "≥ β yes/no". Zero-width windows produce
fail-soft TT entries and frequent cutoffs of their own, so
the probe is far cheaper than a normal child search.

**Reducer R = 3.** Classical fixed reduction. Adaptive
formulas (`R = 3 + depth / 6` and similar) are a 2.x polish
item, not in this release.

**Guards** (every condition must hold; see the comment block
at `search.cpp:513-526`):

1. `allow_null` — never two null moves in a row. The
   recursive null-move call passes `allow_null = false`,
   blocking immediate re-entry. Without this gate, a
   zugzwang chain could pile up.
2. `!disable_alpha_beta` — NMP is a *pruning* technique;
   the analyzer's "no α-β" debug mode skips it.
3. `ply > 0` — never null at the root. The root must
   actually pick a move.
4. `depth ≥ 3` — below this the depth budget after `R + 1`
   reduction is too thin to save meaningful work.
5. `!is_in_check(stm)` — passing while in check is illegal
   in chess; the position would be incoherent.
6. `has_non_pawn_material(stm)` — in king + pawns endgames
   any move can be strictly worse than passing (zugzwang),
   so NMP is unsound. The probe is `helper at search.cpp:420`,
   a constant-time bitboard check.
7. `static_eval ≥ β` — if we are already below β with the
   move, a free reply for the opponent will not push us
   above β; the probe would predictably fail and waste
   work.

**Recorder isolation.** The shadow search runs with
`rec = nullptr`, so the analyzer's tree view never sees the
null-move probe — it would otherwise visualize a fictitious
"opponent moved twice" branch. The same isolation pattern
recurs in LMR.

**Fail-soft return.** When the probe succeeds we return the
*actual* `null_score` rather than `β`. The return is marked
`exact = false` because we visited zero real children: the
node's true value is "≥ score", not the score itself.

**Interaction with PV truncation.** A node that returns via
NMP has `pv.length[p] = 0`, so the parent's PV ends one ply
later. This is the same truncation the TT probe-cuts cause,
and the post-iteration PV-walker in `find_best_impl`
(`search.cpp:822-881`) extends the truncated PV by chasing
TT best-moves until it hits a miss, an illegal move, or a
repeated key. With the walker in place the user-visible PV
is full-depth even when the in-search PV stopped early.

## 2.0.2 — Late move reductions (LMR)

**Idea.** With reasonable move ordering, the first few
sorted moves at any node are far more likely to be the best
than the rest. *Reduce* the depth of the recursive search
on late, ordered-low moves: search them shallower first; if
the shallow probe shows the move was as bad as predicted,
trust that and move on. Only when the probe surprisingly
beats α — meaning the reduction may have hidden a real
improvement — do we re-search at full depth.

The pattern is "trust the ordering, verify only when the
ordering was wrong". On well-ordered trees most reductions
stick; the savings compound multiplicatively with depth.

**Where.** `src/search/search.cpp:685-741`, inside the move
loop in `negamax`. The "reduced probe + conditional
re-search" structure is shared with PVS — the same code
handles both reductions.

**Mechanism.**

```cpp
const bool lmr_candidate =
       stop.enable_lmr
    && !stop.disable_alpha_beta
    && depth >= 3
    && i >= 3
    && buf[i].score < 50'000;       // quiet bucket only

const int R = lmr_candidate ? 1 : 0;

NegamaxResult child = negamax(
    board, depth - 1 - R + ext, child_ply,
    probe_alpha, probe_beta,
    ..., /*rec=*/(R > 0 ? nullptr : rec));
int score = -child.score;

if (R > 0 && !abort && score > alpha) {
    // Re-search at full depth + full window with rec attached.
    child = negamax(board, depth - 1 + ext, child_ply,
                    recurse_alpha, recurse_beta,
                    ..., rec);
    score = -child.score;
}
```

**Reducer R = 1.** A flat one-ply reduction. Engines often
scale R with depth and move index (logarithmic tables); 2.x
keeps it constant, matching NMP's R = 3 in style.

**Candidate gate.** A move qualifies for reduction when
*every* condition holds (`search.cpp:705-710`):

1. `enable_lmr` — analyzer toggle, default on.
2. `!disable_alpha_beta` — a no-pruning debug pass should
   not silently reduce.
3. `depth ≥ 3` — at depth 1 or 2 there is nothing useful
   left to reduce *to*.
4. `i ≥ 3` — keep the first three sorted moves at full
   depth (TT move, killers, top captures usually sit there).
5. `buf[i].score < 50'000` — the "quiet bucket": below the
   promotion-without-capture floor in `score_move`. This
   single test excludes captures (≥ 1'000'000), killers
   (800k / 900k), promotions (≥ 50'000), and the TT move
   (10'000'000) in one comparison. After the history
   heuristic landed (2.0.4), quiet-but-history-hot moves
   sort above genuinely cold quiets, which compounds with
   LMR — only the truly cold tail gets reduced.

**Re-search rule.** A reduced probe that returns
`score > alpha` triggers a re-search at full depth and
recorder attached. We don't gate the re-search on
`score < beta` — a probe that returns ≥ β is also re-searched
to verify before declaring the cutoff. Slightly more
conservative than textbook "lazy PVS" but safer when the
reduction overshot.

**Recorder isolation.** Same pattern as NMP: the reduced
probe runs with `rec = nullptr`. If the probe confirms
fail-low (no re-search), the analyzer tree shows the move
as a leaf with the probe's score, no fictitious children.
If the probe triggers a re-search, the full-depth call
attaches the recorder and the tree gets the confirmed
subtree exactly once.

**Composes with check extensions.** Line 728 reads
`depth - 1 - R + ext`, where `ext = 1` when the move gives
check. A late checking quiet move (a knight check from a
distance) gets reduction *and* extension stacked: net depth
delta is zero. This is intentional — never reduce a checking
move past its full depth; it is exactly the kind of move
where shallow search lies most.

**Composes with PVS.** When PVS is enabled, the probe
window also narrows to `[α, α + 1]` (zero-width) for
non-first moves. LMR contributes the depth reduction; PVS
contributes the window narrowing; both share the same
re-search trigger. See 2.0.5 for PVS specifics.

**Toggle.** `SearchLimits::enable_lmr` (default `false`
to match the rest of the optional flags). The Qt analyzer's
SolveBudget defaults it to `true` and exposes the **LMR**
checkbox alongside NMP, history, aspiration, and PVS.
