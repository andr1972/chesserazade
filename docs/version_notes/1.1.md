# 1.1 — Bitboard alternative

Tagged as **v1.1.0**.

Delivers the 1.1 milestone from HANDOFF §9: a bitboard
implementation of the `Board` interface that passes the same
perft suite as the mailbox and is measurably faster. The
mailbox stays in the tree as the didactic reference.

Shipped across four incremental stages (1.1.1 through 1.1.4),
each its own commit, all collected here.

## What was added

### 1.1.1 — Bitboard primitives + non-slider attack tables
- `include/chesserazade/bitboard.hpp`: `Bitboard = uint64_t`
  under LERF indexing, `bb_of`, `contains`, `set`, `clear`,
  `popcount`, `lsb`, `msb`, `pop_lsb` (wrappers over `<bit>`
  intrinsics), file/rank constant masks.
- `Attacks::king / knight / pawn` — precomputed tables
  initialized once at first use (Meyers singleton, no globals).

### 1.1.2 — `BoardBitboard` skeleton
- New concrete `BoardBitboard : public Board` with 12 piece
  bitboards (`pieces_[color][type]`), per-color occupancy
  cache (`color_`), total occupancy (`occ_`), plus the usual
  side / castling / EP / clocks / zobrist.
- `piece_at`, `from_fen`, FEN setters, `recompute_zobrist`,
  structural equality.
- FEN parser extracted into a shared `parse_fen_fields`
  (`src/io/fen_fields.hpp`) so both board implementations
  use one parse path.
- `make_move` / `unmake_move` still stubs — real logic in
  1.1.3.

### 1.1.3 — Bitboard `make_move` / `unmake_move`
- Three private piece-movement helpers — `remove_piece`,
  `add_piece`, `move_piece` — each updating the piece BB,
  the color BB, the total occupancy, and the Zobrist key in
  lock-step.
- `make_move` mirrors `Board8x8Mailbox::make_move` step-for-
  step; `unmake_move` restores Zobrist wholesale from the
  per-move snapshot rather than re-inverting the XOR chain.
- All six standard perft positions pass at depth 4 on
  BoardBitboard via the existing (abstract-interface) move
  generator.

### 1.1.4 — Slider attacks + bitboard-native generator + speed benchmark
- `Attacks::rook / bishop / queen` — classical loop-based
  ray walks, O(distance-to-blocker). (Magic bitboards are
  an optional post-1.1 polish.)
- `BitboardMoveGenerator` (new
  `src/move_generator/bitboard_move_generator.cpp`): operates
  on whole piece sets. Pawn motion is expressed as board-wide
  shifts (`<< 8` for white push, `<< 7` / `<< 9` for diagonal
  captures with file masks to prevent wrap). Knights / sliders
  / king iterate their piece bitboards with `pop_lsb`, get the
  attack set from `Attacks::X(from, occ)`, mask with
  `~our_occ`, and serialize into captures vs quiet moves.
- `MoveGenerator::generate_legal / generate_pseudo_legal /
  is_in_check / is_square_attacked` each check at their
  entry whether the board is a `BoardBitboard` (via
  `dynamic_cast`) and dispatch to the fast path when so.
  Mailbox callers are unchanged.
- `chesserazade perft` gains `--board mailbox|bitboard`.
- `tools/perft_bench.cpp` now runs every position against
  both implementations and prints a side-by-side speedup
  line. A node-count mismatch flags stderr.
- All six perft positions pass at depth 5 on BoardBitboard.

## Speed comparison (release-with-asserts, depth 5)

Reference machine, single thread, loop-based sliders:

| Position  | Mailbox Mnps | Bitboard Mnps | Speedup |
| --------- | ------------ | ------------- | ------- |
| initial   | 12.0         | 34.4          | **2.9×** |
| kiwipete  | 13.5         | 33.4          | **2.5×** |
| pos3      |  7.7         | 26.3          | **3.4×** |
| pos4      | 12.9         | 31.8          | **2.5×** |
| pos5      | 11.8         | 29.5          | **2.5×** |
| pos6      | 14.6         | 36.0          | **2.5×** |

Depth 6 on the initial position shows a wider gap:
`mailbox 18.4s (6.5 Mnps)  bitboard 3.5s (34.4 Mnps)  x5.3` —
bitboard throughput is roughly constant with depth while
mailbox loses ground to growing working sets.

## Acceptance criteria (HANDOFF §9 1.1)

- [x] `BoardBitboard` implements the `Board` interface with
      the same external contract as `Board8x8Mailbox`.
- [x] Bitboard-idiomatic move generator (`BitboardMoveGenerator`)
      in addition to the classical mailbox one.
- [x] Runtime choice: `chesserazade perft --board
      mailbox|bitboard` picks the implementation; both pass the
      same perft tests.
- [x] Both implementations pass the standard perft suite to
      depth 5.
- [x] `perft` with bitboard is measurably faster than mailbox
      — 2.5–3× on the six standard positions at depth 5 in
      release-with-asserts.
- [x] Mailbox stays in the tree as the reference
      implementation; every test continues to run against
      both.

## What is *not* in 1.1

- **Magic / PEXT bitboards for sliders.** Loop-based ray
  walks are the O(distance) baseline. Magic bitboards get us
  to O(1) per slider lookup and typically another 3-5×
  overall. Out-of-scope for 1.1; a post-release polish pass
  can add them.
- **CLI flag beyond `perft`.** `solve`, `analyze`, `repl`,
  etc. all construct `Board8x8Mailbox` internally. Adding a
  per-command `--board` flag is trivial mechanically but
  offers little over using `perft --board` for the one
  performance-critical path.
- **Bitboard-specific tests beyond perft.** The perft suite
  is the gold standard correctness test for a move generator,
  and it already runs against both boards. Targeted tests for
  slider attacks / pawn shifts would be duplicative.

## Known deviations

- The dispatch in `MoveGenerator` uses `dynamic_cast<const
  BoardBitboard*>(&b)`. This adds a constant-factor overhead
  at each entry point call; in the current benchmarks it is
  not visibly distinguishable from measurement noise. If the
  cost ever becomes material, adding a virtual `kind()` tag
  on `Board` would be the minimal-impact alternative.
- `BitboardMoveGenerator::gen_castling` still does three
  separate `is_square_attacked` calls for the king's from /
  transit / destination squares. A single "attacks-to" query
  over the combined path would be slightly faster; kept clear
  here for parity with the mailbox version.
