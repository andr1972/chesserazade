# 1.1 — Bitboard alternative (+ magic / PEXT polish)

Tagged as **v1.1.0**.

Delivers the bitboard milestone from HANDOFF §9 1.1 — a
second `Board` implementation, a bitboard-native move
generator, a runtime choice via CLI flag, and a measurable
speedup on perft. The stage was broken into five incremental
sub-etaps (1.1.1 through 1.1.5), all collected here.

## What was added

### 1.1.1 — Bitboard primitives + non-slider attack tables
- `include/chesserazade/bitboard.hpp`: `Bitboard = uint64_t`
  under LERF indexing, `bb_of`, `contains`, `set`, `clear`,
  `popcount`, `lsb`, `msb`, `pop_lsb` (wrappers over `<bit>`
  intrinsics), file/rank constant masks.
- `Attacks::king / knight / pawn` — precomputed tables
  initialized once at first use.

### 1.1.2 — `BoardBitboard` skeleton
- New concrete `BoardBitboard : public Board` with 12 piece
  bitboards, per-color occupancy cache, total occupancy, plus
  the usual side / castling / EP / clocks / zobrist.
- `piece_at`, `from_fen`, FEN setters, `recompute_zobrist`,
  structural equality.
- Shared FEN parser (`src/io/fen_fields.hpp`) so both board
  implementations use one parse path.

### 1.1.3 — Bitboard `make_move` / `unmake_move`
- Private helpers `remove_piece` / `add_piece` / `move_piece`
  update piece BB, color BB, occupancy, and Zobrist in
  lock-step.
- Full parity with the mailbox on all six standard perft
  positions to depth 4 via the abstract-interface generator.

### 1.1.4 — Slider attacks + bitboard-native generator
- `Attacks::rook / bishop / queen` (loop-based ray walks,
  O(distance)).
- `BitboardMoveGenerator` — whole-board pawn shifts, `pop_lsb`
  serialization for piece moves, attack-table lookups, `~our`
  masking for capture vs quiet split.
- `MoveGenerator::generate_*` / `is_*` dispatch to the
  bitboard generator via `dynamic_cast` when the board is a
  `BoardBitboard`.
- CLI: `chesserazade perft --board mailbox|bitboard`.
- All six perft positions pass at depth 5 on `BoardBitboard`.

### 1.1.5 — Magic bitboards + PEXT + runtime init chain
**1.1.5a: generator + dispatch** (`src/board/magic.{hpp,cpp}`)
- Brute-force magic finder (sparse random candidates with
  per-rank seeds from the CPW tradition, ~100 ms total).
- Shared attack-table buffer with per-square offsets.
- `Attacks::rook / bishop` now dispatch through a function
  pointer — default is the loop-based reference, magic swaps
  it on init. Loop version remains public as
  `Attacks::rook_loop / bishop_loop` so tests can validate
  parity.

**1.1.5b: persistent constants** (`data/magics.txt`)
- Text format, ~5.8 KB committed: `[rook]` / `[bishop]`
  sections, one line per square with `magic mask shift` in
  hex.
- `init_magic_attacks_from_file` + lookup chain:
  `$CHESSERAZADE_MAGICS` → `$ORIGIN/data/magics.txt` →
  parent/grandparent → compile-time baked
  `CMAKE_SOURCE_DIR/data/magics.txt`.
- `chesserazade magics-gen [--out <path>]` — brute-force
  search + file writer. Default output is
  `./magics.generated.txt`, never `data/magics.txt`, so the
  shipped constants file cannot be clobbered by accident.

**1.1.5c: BMI2 PEXT variant** (optional)
- CMake option `CHESSERAZADE_USE_PEXT=OFF`. When `ON`,
  `src/board/attacks_pext.cpp` joins the library with
  `-mbmi2` localised to that file.
- `_pext_u64(occ, mask)` directly yields the attack-table
  index — no magic multiply, no mask compression.
- Runtime gate: `__builtin_cpu_supports("bmi2")` — a
  PEXT-compiled binary still falls back cleanly on CPUs
  without BMI2.
- `init_slider_attacks()` picks the fastest available path:
  **PEXT → magic-file → magic-gen → loop**. Called from the
  engine's `main()` so every CLI subcommand gets the fast
  path.

## Speed comparison (depth 5, release-with-asserts)

All paths agree on node counts. Mnps = million nodes/sec.

| Position  | mailbox | bb loop         | bb magic        | bb PEXT         |
|-----------|--------:|----------------:|----------------:|----------------:|
| initial   | 12.0    | 33.2 (×2.78)    | 38.4 (×3.21)    | 37.2 (×3.11)    |
| kiwipete  | 13.4    | 33.1 (×2.48)    | 37.2 (×2.78)    | 38.2 (×2.86)    |
| pos3      |  7.6    | 25.6 (×3.37)    | 30.6 (×4.03)    | 30.6 (×4.03)    |
| pos4      | 12.7    | 31.0 (×2.44)    | 34.3 (×2.70)    | 34.6 (×2.72)    |
| pos5      | 11.9    | 29.1 (×2.45)    | 35.8 (×3.00)    | 36.2 (×3.04)    |
| pos6      | 14.6    | 35.7 (×2.45)    | 39.0 (×2.68)    | 38.8 (×2.67)    |

Depth 6 `initial`: mailbox **6.5 Mnps**, bb PEXT **38.4 Mnps**
(**×5.87**).

The bulk of the speedup is mailbox → bb-loop (a proper
bitboard move generator). Magic adds another ~10–15% on top,
and PEXT another 0–3% on top of that. This split is worth
understanding:

- **Loop-based bitboard sliders are already cheap.** Each ray
  step is a single AND with occupancy, early-exit on the
  first blocker. Average ray length in perft is short because
  the board is full of blockers near the piece.
- **Magic is O(1) but hits a ~2 MB table** that doesn't fit
  in L1 cache. A cache miss on the table can cost more than
  a few ray steps in hot registers.
- **Most perft time is not in sliders.** `make_move` /
  `unmake_move` (piece-BB + Zobrist updates) and the
  make+is_in_check+unmake legality filter dominate. Speeding
  up sliders can only shave a fraction of the total.

A heavy-slider workload (open positions, many queens) or an
engine that generates legal moves without the make/unmake
filter would see magic's speedup more clearly. For *this*
engine's architecture, magic is a real but modest polish.

## Acceptance criteria (HANDOFF §9 1.1)

- [x] `BoardBitboard` with the same external contract as
      `Board8x8Mailbox`.
- [x] Bitboard-idiomatic move generator.
- [x] Runtime choice: `chesserazade perft --board
      mailbox|bitboard`.
- [x] Both implementations pass the standard perft suite to
      depth 5.
- [x] Bitboard is measurably faster — 2.5×–3× at depth 5
      without magic, 2.7×–4× with magic, up to ×5.9 at depth 6.
- [x] Mailbox stays in the tree as the reference
      implementation.

## What is *not* in 1.1

- **Pin / check tables.** Legality filtering still goes
  through make + is_in_check + unmake. A pinned-piece cache
  would let the generator skip illegal moves before
  generation, but this is an architectural change outside
  1.1's scope.
- **PEXT as default.** The CMake option is off so the
  out-of-the-box build stays portable to non-BMI2 CPUs. Turn
  on with `-DCHESSERAZADE_USE_PEXT=ON` if your CPU supports
  it (`lscpu | grep bmi2`).
- **Bitboard-specific unit tests beyond perft + magic
  parity.** Perft is the gold-standard correctness probe; a
  targeted set of "did bishop_attacks on c4 return the right
  bitboard" tests would duplicate the check.

## Known deviations

- The `dynamic_cast` in `MoveGenerator::generate_legal` adds
  a constant-factor overhead on every entry. Not
  distinguishable from noise in the current benchmarks; if
  it ever shows up on a profile, a virtual `kind()` tag on
  `Board` would remove it.
- `BoardBitboard::unmake_move` restores the Zobrist wholesale
  from the snapshot instead of inverting the XOR chain.
  Simpler and impossible to drift; skips one extra XOR per
  piece movement during unmake (negligible).
