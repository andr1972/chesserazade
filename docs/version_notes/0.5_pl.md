# 0.5 — Ewaluacja materiałowa + minimax

Otagowane jako **v0.5.0**.

Pierwsza wersja chesserazade, która umie *odpowiadać* na pytania.
0.4 pozwalała załadować PGN i przejść go ponownie; 0.5 umie
powiedzieć „co gra białe tutaj, i jak źle jest?". Wszystko wciąż
klasyczne i sprzed-optymalizacyjne: zwykły negamax, bez alfa-beta.

## Co zostało dodane

### Evaluator (`include/chesserazade/evaluator.hpp`)
*Simplified Evaluation Function* Michniewskiego z Chess
Programming Wiki:

- Wartości bierek w centypionach (P=100, N=320, B=330, R=500,
  Q=900).
- Jedna 64-elementowa tablica piece-square na typ bierki, w
  kolejności LERF od strony białych. Bierka czarna na polu `s`
  czyta `table[s ^ 56]` (sztuczka lustra ranku).
- `evaluate(board)` zwraca z perspektywy **strony na ruchu** —
  dodatnie = dobrze dla tego kto ma ruch. Ta konwencja znaku
  jest tym, co sprawia że negamax działa z jedną pętlą
  neguj-i-porównaj.

Dostępna tylko tablica króla dla debiutu/gry środkowej.
Fazo-zależny wariant (środkowa vs końcowa) to sprawa 0.6+.

### Search (`include/chesserazade/search.hpp`)
Zwykły negamax na stałej głębokości:

```
negamax(board, depth, ply):
    if depth == 0: return evaluate(board)
    legal = legal_moves(board)
    if legal is empty:
        return -MATE_SCORE + ply  if in check else 0
    best = -INF
    for m in legal:
        make(m); s = -negamax(..., depth-1, ply+1); unmake(m)
        best = max(best, s)
    return best
```

Scoring matów: mat dostarczony N plyów od korzenia zwraca
`MATE_SCORE - N`; mat otrzymany zwraca `-(MATE_SCORE - N)`.
Szybsze maty mają wyższy wynik, więc search naturalnie je
preferuje bez jawnego porównywania odległości do mata.

Principal variation wyciągnięta przez triangularną tablicę PV na
stosie — żadnych alokacji wewnątrz negamaxu. `SearchResult` niesie
`best_move`, `score`, `principal_variation`, `nodes` i zegarowy
`elapsed`.

Dwa helpery do interpretacji wyniku:
- `is_mate_score(s)` — true wtedy i tylko wtedy gdy `|s|` leży
  w paśmie wyniku mata (odległość < 1000 plyów od `MATE_SCORE`).
- `plies_to_mate(s)` — odległość w plyach ze znakiem; dodatnie =
  wygrywamy, ujemne = przegrywamy, 0 = nie mat.

### CLI: `solve`
```
chesserazade solve --depth N [--fen <fen>]
```
Drukuje najlepszy ruch (SAN + UCI), wynik (`"+NNN cp"` lub
`"mate in N"` / `"mated in N"`), principal variation jako linię
SAN z numerami ruchów, liczbę węzłów i zegarowy czas.

Przykład:
```
$ chesserazade solve --fen "k7/8/8/1QK5/8/8/8/8 w - - 0 1" --depth 4
best: Kc6  (c5c6)
score: mate in 2
pv:    1. Kc6 Ka7 2. Qb7#
nodes: 3934
time:  7 ms
```

## Kryteria akceptacji (HANDOFF §9 0.5)

- [x] Evaluator z klasycznymi wartościami bierek i tablicami
      piece-square.
- [x] Search ze zwykłym negamaxem, stała głębokość.
- [x] Podkomenda `solve --fen <fen> --depth N`.
- [x] Znajduje wymuszony mat-w-1 (mat na linii podstawowej, wzory
      K+Q) — `plies_to_mate == 1` przy głębokości 2.
- [x] Znajduje wymuszony mat-w-2 (3 ply) — zweryfikowane na
      klasycznej pozycji K+Q vs K z pędzeniem króla do narożnika
      (`1.Kc6 Ka7 2.Qb7#`) przy głębokości 4. Głębokość 4 a nie
      3, bo `negamax` ewaluuje przy `depth == 0` bez sprawdzenia
      terminalności; mat o jeden ply poza liściem byłby
      pominięty przy głębokości 3.

## Czego **nie ma** w 0.5

- **Brak alfa-beta.** To feature-flagowy element 0.6. Głębokość
  rośnie o 1 na poziom gry; przy głębokości 8 silnik już zajmuje
  sekundy na mailboxowym generatorze ruchów.
- **Brak sortowania ruchów, killerów, TT.** Każdy węzeł
  przegląda wszystkie legalne ruchy w kolejności generacji.
- **Brak quiescence search.** Węzły liściowe przyjmują statyczny
  eval jako finalny, więc „zagmatwane" pozycje (wiszące bicia)
  mogą dawać dzikie wyniki na płytkich głębokościach. Quiescence
  przychodzi w 0.8.
- **Brak kontroli czasu.** `--depth N` to jedyne ograniczenie;
  flaga `--time-ms` pojawi się w 0.6 razem z alfa-beta i
  iteracyjnym pogłębianiem.
- **Tablica króla tylko dla gry środkowej.** W końcówce kara za
  wciągniętego króla może spowodować, że silnik odmówi
  aktywacji swojego króla. Interpolacja faz w 0.6+ to naprawi.
- **Mat na depth-0.** Search nie sprawdza ponownie mata przy
  `depth == 0`; zwraca statyczny eval. Praktyczny efekt: żeby
  *wykryć* mat, potrzeba `depth >= (ply do mata) + 1`. To
  klasyczny projekt — silniki zwykle amortyzują koszt przez
  driver iteracyjnego pogłębiania który pojawia się w 0.6.

## Znane odstępstwa

- Brak od HANDOFF §9 0.5.

## Statystyki

- 8 testów evaluatora (symetria pozycji startowej, nierównowaga
  materiałowa, niezmienniczość koloru, preferencje PST, wartości
  bierek).
- 9 testów search (helpery matów, mat-w-korzeniu,
  pat-w-korzeniu, mat-w-1 x 2 pozycje, mat-w-2, spójność PV,
  sentinel depth=0, ruch wygrywający materiał).
- Wszystkie 68 wcześniejszych testów przechodzi.
- **85 / 85 testów przechodzi.**

## Następna: 0.6

Alfa-beta pruning na tym samym szkielecie negamaxu + iteracyjne
pogłębianie + zegarowe `--time-ms` + pierwszoklasowe wyświetlanie
principal variation. Drzewo gwałtownie się kurczy i silnik zaczyna
widzieć głębokość 6-8 w użytecznym czasie.
