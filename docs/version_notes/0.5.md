# 0.5 — Material evaluation + minimax

Tagged as **v0.5.0**.

The first version of chesserazade that can *answer* questions. 0.4
could load a PGN and let you replay it; 0.5 can tell you "what
does white play here, and how bad is it?". Everything is still
classical and pre-optimization: plain negamax, no alpha-beta.

## What was added

### Evaluator (`include/chesserazade/evaluator.hpp`)
Michniewski's *Simplified Evaluation Function* from the Chess
Programming Wiki:

- Centipawn piece values (P=100, N=320, B=330, R=500, Q=900).
- One 64-entry piece-square table per piece type, laid out in
  LERF order from white's perspective. A black piece on square
  `s` reads `table[s ^ 56]` (rank-mirror trick).
- `evaluate(board)` returns from the **side-to-move's**
  perspective — positive = good for whoever is to move. That
  sign convention is what makes negamax work with a single
  negate-and-compare loop.

Only a middlegame king table is provided. A phase-dependent
(middlegame vs endgame) king variant is a 0.6+ concern.

### Search (`include/chesserazade/search.hpp`)
Plain fixed-depth negamax:

```
negamax(board, depth, ply):
    if depth == 0: return evaluate(board)
    legal = legal_moves(board)
    if legal is empty:
        return -MATE_SCORE + ply  if in check else 0
    best = -INF
    for m in legal:
        make(m); s = -negamax(..., depth-1, ply+1); unmake(m)
        best = max(best, s)
    return best
```

Mate scoring: a mate delivered N plies from the root returns
`MATE_SCORE - N`; a mate received returns `-(MATE_SCORE - N)`.
Faster mates score higher, so the search naturally prefers them
without an explicit mate-distance comparison.

Principal variation extracted via a triangular PV table on the
stack — no allocations inside negamax. `SearchResult` carries
`best_move`, `score`, `principal_variation`, `nodes`, and the
wall-clock `elapsed`.

Two helpers for score interpretation:
- `is_mate_score(s)` — true iff `|s|` lies within the mate-score
  band (distance < 1000 plies from `MATE_SCORE`).
- `plies_to_mate(s)` — signed distance in plies; positive = we
  win, negative = we lose, 0 = not a mate.

### CLI: `solve`
```
chesserazade solve --depth N [--fen <fen>]
```
Prints the best move (SAN + UCI), the score (`"+NNN cp"` or
`"mate in N"` / `"mated in N"`), the principal variation as a
SAN line with move numbers, the node count, and the wall-clock
elapsed time.

Example:
```
$ chesserazade solve --fen "k7/8/8/1QK5/8/8/8/8 w - - 0 1" --depth 4
best: Kc6  (c5c6)
score: mate in 2
pv:    1. Kc6 Ka7 2. Qb7#
nodes: 3934
time:  7 ms
```

## Acceptance criteria (HANDOFF §9 0.5)

- [x] Evaluator with classical piece values and piece-square
      tables.
- [x] Search with plain negamax, fixed depth.
- [x] `solve --fen <fen> --depth N` subcommand.
- [x] Finds forced mate-in-1 (back-rank mate, K+Q mate patterns)
      — `plies_to_mate == 1` at depth 2.
- [x] Finds forced mate-in-2 (3 plies) — verified on the classic
      K+Q vs K corner-drive position (`1.Kc6 Ka7 2.Qb7#`) at
      depth 4. Depth 4 rather than 3 because `negamax` evaluates
      at `depth == 0` without a terminal check; a mate one ply
      beyond the leaf would be missed at depth 3.

## What is *not* in 0.5

- **No alpha-beta.** That is the headline feature of 0.6. Depth
  grows by 1 per level of play; at depth 8 the engine already
  takes seconds on a mailbox move generator.
- **No move ordering, no killers, no TT.** Every node explores
  all legal moves in generation order.
- **No quiescence search.** Leaf nodes take the static eval as
  final, so "noisy" positions (hanging captures) can produce
  wild scores at shallow depth. Quiescence arrives in 0.8.
- **No time control.** `--depth N` is the only throttle; a
  `--time-ms` flag shows up in 0.6 alongside alpha-beta and
  iterative deepening.
- **King PST is middlegame-only.** Endgame tucked-king penalty
  can make the engine refuse to activate its king in late
  endings. A 0.6+ phase interpolation fixes this.
- **Mate-at-depth-0.** The search does not re-check for mate at
  `depth == 0`; it returns the static eval. Practical effect:
  to *detect* a mate you need `depth >= (plies to mate) + 1`.
  This is the classical design — engines usually amortize the
  cost via the iterative-deepening driver that ships in 0.6.
