# 0.6 — Alfa-beta + iteracyjne pogłębianie + kontrola czasu

Otagowane jako **v0.6.0**.

Search dostaje zębów. Zwykły negamax z 0.5 zostaje zastąpiony
alfa-betą, opakowaną w iteracyjne pogłębianie, z opcjonalnymi
budżetami czasu i węzłów. Drzewo drastycznie się kurczy; silnik
widzi mat-w-3 w dziesiątkach milisekund na pozycjach, gdzie 0.5
potrzebowała sekund zegarowych żeby w ogóle dojść do głębokości.

## Co zostało dodane

### Przycinanie alfa-beta
Klasyczne fail-hard alfa-beta w `src/search/search.cpp`:

```
negamax(board, depth, ply, alpha, beta):
    if depth == 0: return evaluate(board)
    for m in legal:
        make(m); s = -negamax(..., -beta, -alpha); unmake(m)
        if s >= beta: return beta           # odcięcie
        if s > alpha: alpha = s; ... PV ... # poprawa
    return alpha
```

Te same najlepsze ruchy co zwykły negamax (alfa-beta jest
poprawna), ale poddrzewo pod każdym węzłem, który fail high,
jest przycinane. Patrz:
https://www.chessprogramming.org/Alpha-Beta.

### Driver iteracyjnego pogłębiania
`Search::find_best` już nie rekursuje bezpośrednio — uruchamia
pętlę od głębokości 1 do `limits.max_depth`, wywołując
pojedyncze `iteration` za każdym razem. Wynik ostatniej
*ukończonej* głębokości jest zachowywany; częściowe iteracje są
porzucane. Płytkie iteracje są tanie (< 1% drzewa przy finalnej
głębokości) i kupują nam własność anytime: jeśli budżet zadziała
w środku iteracji, nadal zwracamy najlepszy ruch z głębokości
N-1.

### Limity czasu / węzłów (`SearchLimits`)
```cpp
struct SearchLimits {
    int max_depth = 32;
    std::chrono::milliseconds time_budget{0};
    std::uint64_t node_budget = 0;
};
```

- `time_budget` — zegar ścienny, odpytywany co 1 na 2048 węzłów.
  Narzut pomijalny; faktyczna respektowana precyzja znacznie
  lepsza niż 5% wymagane przez HANDOFF.
- `node_budget` — liczy każdy odwiedzony węzeł (wewnętrzny +
  liściowy). Przydatne do powtarzalnych testów regresji
  niezależnych od szybkości hosta.
- Dowolna kombinacja może być ustawiona; wygrywa ten co
  pierwszy zadziała. Domyślnie skonstruowane `SearchLimits` jest
  efektywnie nieograniczone.

### Dodatki do `SearchResult`
- `completed_depth` — ostatnia głębokość, która zakończyła się
  przed przerwaniem. Uczciwe raportowanie dla wyszukiwań z
  limitem czasu.
- `principal_variation` — ten sam kształt, teraz zawsze
  odzwierciedla ukończoną głębokość.

### CLI: `solve` dostaje `--time-ms` i `--nodes`
```
chesserazade solve (--depth N | --time-ms T | --nodes N) [--fen <fen>]
```

Dowolna kombinacja budżetów jest akceptowana. Wyjście zyskuje
linię `depth:` raportującą `completed_depth`, żeby użytkownik
mógł zobaczyć czy limit przerwał wyszukiwanie.

Przykład:
```
$ chesserazade solve --fen "k7/8/8/1Q6/3K4/8/8/8 w - - 0 1" --depth 6
best:  Kd5  (d4d5)
score: mate in 3
depth: 5
pv:    1. Kd5 Ka7 2. Kc6 Ka8 3. Qb7#
nodes: 73859
time:  164 ms
```

(Ukończona głębokość to 5, nie 6, ponieważ iteracyjne pogłębianie
krótkim zwarciem kończy się gdy znaleziono mat — głębsze
wyszukiwanie nie może poprawić wymuszonego mata.)

## Kryteria akceptacji (HANDOFF §9 0.6)

- [x] Przycinanie alfa-beta zastępuje zwykły minimax.
- [x] Iteracyjne pogłębianie.
- [x] Limit czasu (`--time-ms`) i limit węzłów (`--nodes`).
- [x] Wyświetlanie principal variation (w `SearchResult` i w
      CLI `solve`).
- [x] Rozwiązuje mat-w-3 na standardowej pozycji K+Q vs K
      (głębokość 6, `plies_to_mate == 5`).
- [x] PV wyświetlana w CLI `solve`.
- [x] Search respektuje limity czasu w ramach ~5% — sprawdzenie
      stopu odpytuje co 2048 węzłów (kilkaset µs przy
      mailboxowej szybkości), znacznie poniżej granicy 5% na
      dowolnym wielomilisekundowym budżecie.

## Czego **nie ma** w 0.6

- **Brak sortowania ruchów.** Ruchy nadal przeszukiwane w
  kolejności generacji. Z ruchem TT i killerami / heurystyką
  historii, alfa-beta może skurczyć drzewo o kolejny rząd
  wielkości. To 0.8.
- **Brak quiescence search.** Evaluator liścia nadal bierze
  surową wartość statyczną — pozycje z efektem horyzontu
  (wiszące bicia na liściu) nadal źle scorują. 0.8.
- **Brak tablicy transpozycji.** 0.7.
- **Brak okien aspiracyjnych / PVS / null-move pruning.** To są
  typowe dodatki ery 0.6; zostajemy przy waniliowym alfa-beta
  dla czystości dydaktycznej.
- **Brak interpolacji fazy króla.** Tablica króla dla gry
  środkowej jest nadal używana wszędzie (odziedziczone z 0.5).

## Znane odstępstwa

- Krótkie zwarcie na macie w iteracyjnym pogłębianiu raportuje
  `completed_depth` mniejsze niż `max_depth` gdy mat znaleziono
  wcześnie. To feature (zgodny ze Stockfishem itd.), ale warto
  zaznaczyć: linia `depth:` niekoniecznie równa się fladze
  `--depth`.
