# 0.6 — Alpha-beta + iterative deepening + time control

Tagged as **v0.6.0**.

The search grows teeth. 0.5's plain negamax is replaced by
alpha-beta, wrapped in iterative deepening, with optional time
and node budgets. The tree shrinks dramatically; the engine can
see mate-in-3 in tens of milliseconds on positions where 0.5
needed seconds of wall clock even to reach the depth.

## What was added

### Alpha-beta pruning
Classical fail-hard alpha-beta in `src/search/search.cpp`:

```
negamax(board, depth, ply, alpha, beta):
    if depth == 0: return evaluate(board)
    for m in legal:
        make(m); s = -negamax(..., -beta, -alpha); unmake(m)
        if s >= beta: return beta           # cutoff
        if s > alpha: alpha = s; ... PV ... # improve
    return alpha
```

Same best moves as plain negamax (alpha-beta is sound), but the
sub-tree under each node that fails high is pruned. See
https://www.chessprogramming.org/Alpha-Beta.

### Iterative deepening driver
`Search::find_best` no longer recurses directly — it runs a loop
from depth 1 to `limits.max_depth`, calling a single-depth
`iteration` each time. The result of the last *completed* depth
is kept; partial iterations are discarded. The shallow
iterations are cheap (< 1% of the tree at the final depth) and
buy us the anytime property: if the budget fires mid-iteration,
we still return the best move from depth N-1.

### Time / node limits (`SearchLimits`)
```cpp
struct SearchLimits {
    int max_depth = 32;
    std::chrono::milliseconds time_budget{0};
    std::uint64_t node_budget = 0;
};
```

- `time_budget` — wall clock, polled on a 1-in-2048 node cadence.
  Overhead is negligible; actual respected precision is well
  within the 5% the HANDOFF asks for.
- `node_budget` — counts every visited node (interior + leaf).
  Useful for reproducible regression tests that need to be
  independent of host speed.
- Any combination may be set; first to trigger wins. A
  default-constructed `SearchLimits` is effectively unlimited.

### `SearchResult` additions
- `completed_depth` — the last depth that finished before an
  abort. Honest reporting for time-limited searches.
- `principal_variation` — unchanged in shape, now always
  reflects the completed depth.

### CLI: `solve` gets `--time-ms` and `--nodes`
```
chesserazade solve (--depth N | --time-ms T | --nodes N) [--fen <fen>]
```

Any combination of budgets is accepted. Output gains a `depth:`
line reporting `completed_depth` so the user can tell whether
the limit interrupted the search.

Example:
```
$ chesserazade solve --fen "k7/8/8/1Q6/3K4/8/8/8 w - - 0 1" --depth 6
best:  Kd5  (d4d5)
score: mate in 3
depth: 5
pv:    1. Kd5 Ka7 2. Kc6 Ka8 3. Qb7#
nodes: 73859
time:  164 ms
```

(Completed depth is 5, not 6, because iterative deepening
short-circuits once a mate score is found — a deeper search
cannot improve on a forced mate.)

## What is *not* in 0.6

- **No move ordering.** Moves are still searched in generation
  order. With a TT move and killers / history, alpha-beta can
  shrink another order of magnitude. That's 0.8.
- **No quiescence search.** Leaf evaluator still takes the raw
  static value — horizon-effect positions (hanging captures at
  the leaf) still mis-score. 0.8.
- **No transposition table.** 0.7.
- **No aspiration windows / PVS / null-move pruning.** These are
  the usual 0.6-era extras; we stay with vanilla alpha-beta for
  teaching clarity.
- **No king-phase interpolation.** The middlegame king table is
  still used everywhere (inherited from 0.5).

## Known deviations

- The mate-in-N short-circuit inside iterative deepening reports
  a `completed_depth` smaller than `max_depth` when a mate is
  found early. This is a feature (matches Stockfish et al.) but
  is worth flagging: the `depth:` line does not necessarily
  equal the `--depth` flag.
