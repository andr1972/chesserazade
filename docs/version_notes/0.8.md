# 0.8 — Move ordering + quiescence + PuzzleSolver

Tagged as **v0.8.0**.

Three textbook additions turn 0.7's "correct but blind" alpha-
beta into a recognizably competent search. Tactics that were
missed at shallow depth get found; the TT hit rate jumps from
~2% to the 30–50% range; a dedicated puzzle entry point
finishes HANDOFF's 0.x "find mates" track.

## What was added

### Move ordering
`src/search/search.cpp` — new `score_move` / `score_and_sort`
helpers and a per-ply `KillerTable`.

Priorities, highest first:
```
  10'000'000   TT move from the probe at this node
   1'000'000 + MVV-LVA  — captures (10 × victim − attacker)
     900'000   first killer at this ply
     800'000   second killer at this ply
      50'000 + promotion value — quiet promotions
           0   plain quiet moves
```

Killers are updated on every beta cutoff whose causing move is
**not** a capture (captures already sort high via MVV-LVA —
spending a scarce killer slot on them would be redundant). The
`KillerTable` is allocated once per search and persists across
iterative-deepening iterations.

References:
- https://www.chessprogramming.org/Move_Ordering
- https://www.chessprogramming.org/MVV-LVA
- https://www.chessprogramming.org/Killer_Heuristic

### Quiescence search
When the main negamax reaches `depth == 0`, it now calls
`quiesce(alpha, beta)` instead of `evaluate(board)`:

```
quiesce(board, α, β):
    stand_pat = evaluate(board)
    if stand_pat ≥ β: return β
    if stand_pat > α: α = stand_pat
    for capture in legal captures, MVV-LVA-ordered:
        make; s = -quiesce(-β, -α); unmake
        if s ≥ β: return β
        if s > α: α = s
    return α
```

Only captures (including en-passant and promotion-captures) are
explored. Non-captures never escape the horizon, so the recursion
terminates in O(material) steps. This closes the classical
*horizon effect*: a search ending in the middle of a capture
sequence no longer returns the pre-capture static eval as final.

Reference: https://www.chessprogramming.org/Quiescence_Search

### PuzzleSolver (`include/chesserazade/puzzle_solver.hpp`)
Thin wrapper over `Search::find_best` that picks an appropriate
depth for a "mate in N" puzzle:

```cpp
SearchResult PuzzleSolver::solve_mate_in(Board& b, int moves,
                                         TranspositionTable* tt = nullptr);
```

Depth choice is `2N + 1` plies:
- `2N − 1` is the true tree depth of a mate-in-N line (white's
  Nth move delivers mate; the mated side has no legal reply at
  ply `2N − 1`).
- Our negamax short-circuits to `quiesce` at `depth == 0` without
  a terminal check, so we add one ply of headroom.
- Iterative deepening stops early on a mate, so the extra plies
  are free on found mates.

### CLI: `solve --mate-in N`
`solve` gains a puzzle mode:

```
$ chesserazade solve --fen "k7/8/8/1Q6/3K4/8/8/8 w - - 0 1" --mate-in 3
best:  Kd5  (d4d5)
score: mate in 3
depth: 6
pv:    1. Kd5 Ka7 2. Kc6 Ka8 3. Qb7#
nodes: 41433
tt:    28105 probes, 10546 hits (37.5%)
time:  119 ms
```

On a failed puzzle (no mate in N within the depth), `solve`
prints `no mate in N found (best score …)` and exits with 1 —
useful from scripts and future REPL integration.

## What is *not* in 0.8

- **History heuristic.** A per-(piece, to-square) counter
  incremented on beta cutoff, used to order remaining quiet
  moves — the classical third move-ordering tier. Skipped here
  because killers + MVV-LVA + TT already give the node-count
  reduction HANDOFF asked for; adding history is a one-file
  tweak for 0.9.
- **Check extensions / null-move pruning / LMR.** Out of scope
  for the 0.x line, per the educational mandate.
- **Checks in quiescence.** We only search captures. "Noisy
  non-captures" (checks) would add tactical depth but also
  explode the tree; standard practice is to gate them behind a
  fail-soft check evasion test, which is a 1.x concern.
- **History-heuristic-based LMR**, **Singular extensions**,
  **multi-PV** — none of these.

## Known deviations

- The TT integration test (`Search + TT: same best move, fewer
  nodes`) now asserts only on the *score*, not the *best move*.
  With move ordering and TT hints in play, tied-score roots can
  resolve to a different (but equally good) first move between
  runs. Alpha-beta remains sound — the score stays identical.
