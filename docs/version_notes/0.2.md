## What was added

### Move â€” MoveKind + piece context
`include/chesserazade/move.hpp` grew three fields:

- `MoveKind` enum â€” eight mutually-exclusive kinds:
  `Quiet`, `DoublePush`, `KingsideCastle`, `QueensideCastle`,
  `Capture`, `EnPassant`, `Promotion`, `PromotionCapture`.

  beyond relocating the piece on `from`.
- `moved_piece` and `captured_piece` â€” the piece data the

  not have to re-derive it from the post-move board.

0.1 test cases and the FEN layer are unchanged.

### Board â€” make_move / unmake_move
`include/chesserazade/board.hpp` adds two pure-virtual
mutators:

```cpp
virtual void make_move(const Move&) noexcept = 0;
virtual void unmake_move(const Move&) noexcept = 0;
```

`Board8x8Mailbox` implements them with a **snapshot history**:

- `StateSnapshot { ep_square, castling, halfmove_clock }`
  is pushed on every `make_move`. These three fields are not
  recoverable from the post-move board, so they must be
  remembered explicitly.
- `unmake_move` flips the side first, pops the snapshot,
  then restores pieces per `MoveKind`. The caller must pass
  the same `Move` object that was given to the matching
  `make_move`.

Castling rights are updated on: king move (both rights
revoked for that color), rook move from a home corner (the
matching right revoked), and capture *onto* a home corner
(the matching right revoked â€” covers capturing the opposing
rook). En-passant target is cleared every move and set only
by `DoublePush`.

### Move generator (`include/chesserazade/move_generator.hpp`,
### `src/move_generator/move_generator.cpp`)

Pipeline: **pseudo-legal enumeration + legality filter**.

- `MoveList` â€” fixed-capacity `array<Move, 256>` with an
  index counter and `begin`/`end`. 256 is comfortable
  headroom over the theoretical 218-move maximum.
- `generate_pseudo_legal(const Board&)` â€” iterate all 64
  squares, dispatch by piece type:
  - **Pawn:** single push, double push (from rank 2/7,
    both squares clear), diagonal captures, en-passant
    (matches the board's EP target), four-way promotions
    and promotion-captures (Queen / Rook / Bishop / Knight).
  - **Knight:** eight jump offsets.
  - **Bishop / Rook / Queen:** ray walking using explicit
    rank/file integers (not raw square offsets) so that
    rays cannot wrap around the board edge.
  - **King:** eight adjacent offsets + castling via
    `gen_castling`.
- `gen_castling` requires the right, the squares between
  king and rook empty, and the king's *from*, *through*, and
  *to* squares unattacked (the B-file is only checked for
  emptiness, not attack, per the standard rules).
- `generate_legal(Board&)` â€” the board is mutable because
  legality filtering needs `make_move` / `is_in_check` /
  `unmake_move`. After `make_move` the side flips, so the
  code uses the pre-move side when asking
  `is_in_check(b, mover)`.
- `is_square_attacked(Board, Square, attacker_color)` â€” the
  classical "look from the target" technique. Rays are cast
  outward from `sq`; if the first piece found along a ray
  (or at a knight/pawn offset) is the matching enemy piece
  type, the square is attacked. This avoids the cost of
  generating all enemy moves.
- `is_in_check(Board, side)` scans the 64 squares for the
  king of `side` and asks `is_square_attacked` with the
  opposite color.

Nothing here is templated; `MoveGenerator` is a class with
only static methods (there is no per-call state) and
`MoveGenerator() = delete` so it cannot be instantiated.

### CLI
`src/cli/` gains two subcommands:

- `chesserazade moves [--fen <fen>]` â€” prints every legal
  move as UCI, one per line, **sorted**, with a final
  `total: N` line. Sorted output makes scripted diffing
  against a reference generator stable.
- `chesserazade perft --depth N [--fen <fen>] [--divide]` â€”
  recursive perft count. `--divide` prints
  `<uci>: <nodes>` for each root move (matching Stockfish's
  `go perft` format), then the totals.

## Known deviations

- The depth-6 perft for the initial position is verified
  only by hand. This matches the test-policy comment in
  `tests/test_perft.cpp`: automated tests use only
  depth/node limits, no wall-clock timeouts, and a mailbox
  generator is slow enough that depth 6 would dominate CI.
- `MoveList` stores moves by value in a fixed-size array
  rather than a heap-allocated vector. This is a classical
  choice â€” deep search allocates a `MoveList` per node, and
  a stack array is both cache-friendly and trivially
  recycled.
